# 前端工作流图可视化与动画交互设计

日期：2026-04-15

## 1. 文档目标

这份文档是对“模板驱动的受约束 DAG 工作流设计”的前端细化补充，专门回答下面这些问题：

- 前端默认应该展示哪一张图
- 任务运行时图如何实时更新
- 动画应该表达什么语义
- 点击节点后应该看到什么信息
- 后端需要提供什么图数据契约
- 在移动端、低性能设备和低动效模式下如何降级

这份文档不负责定义工作流节点本身，也不重新定义模板、依赖图和产物协议。那些内容以上层工作流 spec 为准，这里只关注前端呈现与交互。

## 2. 背景与定位

在新的工作流模型下，系统底层已经不再是简单的线性 stage 列表，而是：

- 一张受约束的能力图
- 一组技术组合模板
- 每次任务运行时的一张执行子图

因此，前端如果仍然只展示“线性进度条 + 文本列表”，会出现几个问题：

- 用户不知道当前模板到底会执行哪些节点
- 用户无法直观看到 OCR、字幕擦除等支线是否参与本次执行
- 节点间依赖和 fallback 行为难以解释
- `cached`、`partial_success`、`optional failure` 这些状态很难表达清楚

所以，工作流图不再是一个“可选的炫酷组件”，而是新的任务执行模型在前端的主要解释界面。

## 3. 设计原则

这份可视化设计坚持以下原则：

1. **默认展示执行子图，不展示完整能力图**
   用户首先需要理解“这次任务会跑什么”，而不是“系统理论上能跑什么”。

2. **动画服务于状态表达**
   动画必须帮助用户理解运行状态，而不是为了“更炫”。

3. **图是主监控界面，文本是辅助界面**
   任务运行中，图应持续可见；日志、配置和产物详情通过钻取展开。

4. **布局必须稳定**
   同一模板在不同任务上的图布局应尽量一致，便于用户建立长期心智模型。

5. **低动效和低性能场景必须可用**
   动画是增强，不是功能前提。

## 4. 图对象定义

前端需要明确区分三种图对象，避免概念混用。

### 4.1 能力图（Capability Graph）

定义：

- 系统支持的完整节点与依赖关系图

用途：

- 帮助开发者理解系统能力边界
- 用于说明“有哪些节点”和“它们理论上如何连接”

前端策略：

- 第一版不默认展示
- 可以作为二级视图存在
- 不作为用户主流程入口

### 4.2 模板预览图（Template Preview Graph）

定义：

- 用户选中某个模板后，尚未开始执行前的 resolved execution subgraph

用途：

- 帮助用户理解这个模板会执行哪些节点
- 在创建任务页中解释模板和配置的关系

前端策略：

- 作为“新建任务”或“任务配置确认”页面的主要可视化
- 可以随着模板或 policy override 改变而即时更新

### 4.3 运行执行图（Runtime Execution Graph）

定义：

- 某个实际任务在运行中的执行子图，带有实时状态

用途：

- 作为任务详情页的主要监控界面
- 让用户理解当前在哪个节点、后续会跑什么、哪些节点被缓存复用、哪些节点失败

前端策略：

- 这是默认展示的图
- 任务运行中持续可见
- 任务完成后仍保留，作为任务回溯界面

## 5. 默认展示策略

默认展示策略明确如下：

- 在任务创建阶段：
  - 默认展示模板预览图
- 在任务详情页：
  - 默认展示运行执行图
- 不默认展示完整能力图

如果后续需要完整能力图，建议放在：

- 设置页
- 帮助页
- 或任务详情里的“查看完整能力图”次级开关

这样主界面始终围绕“本次任务实际发生什么”展开。

## 6. 页面与信息架构

### 6.1 新建任务页

图的角色：

- 模板解释器
- 参数影响预览器

应展示内容：

- 本次任务选择的模板
- 模板展开后的执行子图
- 节点分组
- required / optional 标记
- 当前交付策略将使用哪些资产线

不需要展示：

- 实时进度
- 日志内容
- 产物列表

### 6.2 任务详情页

图的角色：

- 运行监控主界面
- 节点详情入口

应展示内容：

- 当前执行图
- 整体工作流状态
- 当前活跃节点
- 节点实时状态变更
- 节点钻取详情

不建议：

- 图在运行时被详情 panel 替代
- 图折叠后默认只显示文本列表

### 6.3 仪表盘页

图的角色：

- 轻量概览图

建议：

- 对于活跃任务卡片，只显示简化版执行图
- 不在仪表盘上塞入完整可钻取 inspector
- 点击卡片进入任务详情页看完整交互

## 7. 图布局设计

### 7.1 布局类型

必须使用**固定分层 DAG 布局**，不能使用 force-directed 布局。

原因：

- force 布局每次都抖动，不利于长期记忆
- 运行时动画叠加 force 布局，会产生明显视觉噪音
- 节点数不大，没必要用物理模拟

### 7.2 Lane 分组

图按四条 lane 展示：

1. 音频主干
2. OCR 字幕线
3. 视频净化线
4. 交付线

每条 lane 都应有稳定位置，不因任务状态变化而改位。

### 7.3 节点相对位置

节点在同一模板内应尽量固定顺序：

- 音频主干始终按 `stage1 -> task-a -> task-b -> task-c -> task-d -> task-e`
- OCR 字幕线始终按 `ocr-detect -> ocr-translate`
- 视频净化线始终按 `ocr-detect -> subtitle-erase`
- 交付节点固定在最右侧或最后一列

这样不同任务之间切换时，用户不需要重新学习布局。

### 7.4 边布局

边使用 SVG 或其他确定性向量渲染方式。

建议：

- 普通边采用柔和曲线或正交折线
- 当前活跃边支持方向性感知
- 被阻塞边应有明显视觉区别

不建议：

- 复杂自交线
- 动态路由导致边位置频繁漂移

## 8. 节点视觉设计

### 8.1 节点内容

每个节点卡片至少包含：

- 节点名称
- 节点组别或图标
- 当前状态
- 简要进度信息

运行中或完成后可附加：

- 已耗时
- cache 标记
- optional 标记

### 8.2 必要的视觉区分

必须视觉区分以下概念：

- required 与 optional
- pending 与 cached
- running 与 active-path
- succeeded 与 partial_success 所属节点
- failed 与 blocked

这几个状态如果只靠颜色一种手段表达，会不够稳。建议组合使用：

- 颜色
- 图标
- 边框样式
- 轻动效
- 文本标签

### 8.3 推荐状态语义

建议语义如下：

- `pending`
  - 中性灰
  - 低强调
- `running`
  - 蓝色主高亮
  - 有轻微动态边流
- `cached`
  - 紫色或青紫色
  - 明确有“已复用”标签
  - 动效比 running 更弱
- `succeeded`
  - 绿色稳定完成态
- `failed`
  - 红色高优先级失败态
- `skipped`
  - 去强调、低饱和状态
- `blocked`
  - 只用于边或下游节点说明被上游失败阻塞

## 9. 动画设计

### 9.1 总原则

动画只承担三类任务：

1. 建立层级与入场秩序
2. 表达当前运行态
3. 强调状态切换

不承担：

- 长时间吸引注意力
- 持续制造背景噪音
- 依赖物理模拟来“显得高级”

### 9.2 初始加载动画

推荐行为：

- 节点做短时 stagger reveal
- 连线稍晚于节点出现
- 总时长保持克制

目标：

- 让用户快速看懂图的结构
- 不让页面加载显得生硬

### 9.3 运行中动画

运行中图的关键动效应该包括：

- 当前节点主高亮
- 当前路径边有方向性感知
- 整体进度平滑更新

推荐行为：

- 运行中节点：
  - 微弱呼吸光或边框脉冲
- 活跃边：
  - 缓慢、有方向性的流动效果
- 非活跃节点：
  - 不产生多余动态

这样用户一眼就能看到：

- 现在卡在哪个节点
- 当前路径从哪来、往哪去

### 9.4 缓存命中动画

`cached` 不能看起来像“还没执行”。

推荐行为：

- 使用一次性、短时的确认动效
- 然后进入稳定低动效状态

不建议：

- 对 cached 节点维持持续呼吸动画

因为它不是当前焦点，不能和 running 抢注意力。

### 9.5 成功与失败动画

成功：

- 用轻量 completion 动效
- 例如边框/图标快速收敛，不要长期闪烁

失败：

- 当前活跃动画立即停住
- 失败节点和相关边成为最显眼的对象
- 下游 blocked 节点进入被阻塞状态

目标：

- 用户能快速定位失败点
- 不是被整张图的其它动效干扰

### 9.6 Partial Success 的整体表达

`partial_success` 是整体工作流级语义，不能只通过单个节点颜色暗示。

建议同时在两层体现：

- 顶部任务状态条
- 图上 optional 失败节点的状态

这样用户可以理解：

- 主链成功了
- 但某些可选能力没有成功

## 10. 可钻取交互设计

### 10.1 交互目标

用户点击图上的节点后，应立即回答这些问题：

- 这个节点是干什么的
- 它现在是什么状态
- 它是 required 还是 optional
- 它用了哪些输入
- 它产生了哪些输出
- 为什么失败
- 我接下来应该看哪里

### 10.2 推荐交互容器

推荐使用右侧 drawer 或侧边详情面板，而不是居中 modal。

原因：

- 任务运行时图仍然保持可见
- 节点之间切换更连贯
- 详情不会打断整体上下文

### 10.3 节点详情面板内容

第一版应至少包含：

- 节点名称
- 所属 lane / group
- required 或 optional
- 当前状态
- 当前进度
- 开始时间 / 已耗时
- 执行摘要
- 输入产物引用
- 输出产物引用
- manifest 链接
- log 链接
- 失败摘要

可选增强项：

- 最近状态变更时间
- 节点级配置快照
- 节点级质量报告入口

### 10.4 节点切换行为

当用户点击不同节点时：

- 详情面板内容应平滑切换
- 图本身不应重新布局
- 当前选中节点要有明确视觉选中态

## 11. 后端图数据契约

前端不应该自己从 `workflow-manifest.json` 或 `workflow-status.json` 中反推出图结构。

后端应提供专门的图数据契约。

### 11.1 推荐结构

```json
{
  "workflow": {
    "task_id": "task-123",
    "template_id": "asr-dub+ocr-subs",
    "status": "running",
    "selected_policy": {
      "video_source": "clean_if_available",
      "audio_source": "both",
      "subtitle_source": "both"
    }
  },
  "nodes": [
    {
      "id": "task-c",
      "label": "Task C",
      "group": "audio-spine",
      "required": true,
      "status": "running",
      "progress_percent": 67,
      "summary": "batch 3/4",
      "manifest_path": "/path/to/task-c-manifest.json",
      "log_path": "/path/to/task-c.log"
    }
  ],
  "edges": [
    {
      "from": "task-b",
      "to": "task-c",
      "state": "active"
    }
  ]
}
```

### 11.2 边状态

建议边状态支持：

- `inactive`
- `active`
- `completed`
- `blocked`

这样动画和视觉语义就有稳定数据来源。

### 11.3 适用场景

这份契约需要同时支持：

- 模板预览图
- 运行执行图
- 已完成任务的回放式查看

## 12. 实时更新模型

### 12.1 更新原则

图更新应以“状态增量变化”为核心，而不是每次整图重建。

原因：

- 整图重建会导致布局闪动
- 动画容易断裂
- 用户更难保持空间记忆

### 12.2 推荐数据更新策略

推荐：

- 任务创建时获取一次完整 resolved graph
- 运行期间通过 SSE 或轮询推送节点状态增量
- 前端在本地 store 中合并节点状态

### 12.3 不建议的策略

不建议：

- 每次状态更新都返回完整重新排序图
- 前端每次都销毁并重建所有节点组件

## 13. 组件结构建议

建议新增或细化以下组件：

- `WorkflowGraph`
  - 图容器与布局层
- `WorkflowLane`
  - lane 分组容器
- `WorkflowNodeCard`
  - 节点卡片
- `WorkflowEdge`
  - 边渲染与边动画
- `WorkflowLegend`
  - 状态和语义图例
- `WorkflowNodeDrawer`
  - 节点详情 drawer

建议配套 hooks：

- `useWorkflowGraph`
  - 获取图数据
- `useWorkflowRuntimeUpdates`
  - 订阅状态更新
- `useSelectedWorkflowNode`
  - 管理节点选中状态

## 14. 性能与可访问性

### 14.1 性能边界

第一版性能目标不是支持超大图，而是：

- 少量节点下稳定、清晰、不卡顿
- 动画不影响任务监控体验

建议：

- 使用 SVG 或确定性 Canvas/SVG 混合渲染
- 控制动效频率和数量
- 避免复杂 filter 和高代价阴影动画

### 14.2 动效降级

必须支持：

- `prefers-reduced-motion`
- 低性能设备的低动效模式

降级后仍应保留：

- 状态颜色
- 图标
- 文本标签
- 选中与失败强调

### 14.3 移动端降级

移动端不强求完整桌面图体验。

建议：

- 小屏时优先展示简化图
- 或直接降级成 lane 列表 + 节点状态卡片

但仍应保留：

- 当前活跃节点
- required / optional 信息
- 节点可钻取详情

## 15. 非目标

这份前端 spec 当前不负责：

- 设计完整能力图浏览器
- 实现用户自定义拖拽编排
- 实现多用户协作批注
- 实现复杂时间回放动画
- 决定最终动画库的唯一选型细节

它只负责把“当前模板/任务的执行子图”定义清楚。

## 16. 验收标准

这块设计完成后，前端实现至少要满足：

1. 用户在创建任务时能看到模板展开后的执行子图。
2. 用户在任务运行时能持续看到执行图，而不是只看文本进度。
3. 图能稳定表达：
   - `pending`
   - `running`
   - `cached`
   - `succeeded`
   - `failed`
   - `skipped`
4. `partial_success` 有独立整体语义表达。
5. 点击任意节点后，可以打开节点详情面板。
6. 后端提供专门的图数据契约，前端无需自行猜图。
7. 在 reduced motion 模式下仍然可用。
8. 小屏设备有合理降级方案。

## 17. 最终建议

第一版前端工作流图应当被定义为：

- 一个默认展示“模板执行子图 / 任务执行子图”的可视化组件
- 一个运行中持续可见的主监控界面
- 一个支持节点钻取的解释型交互界面
- 一个以稳定分层布局和轻量状态动画为核心的产品组件

不要把它做成：

- 完整能力图浏览器优先
- force graph 演示组件
- 只读静态图
- 动画很多但无法帮助理解执行状态的装饰模块

这个组件的职责，是帮助用户理解“这次任务为什么这么跑、现在跑到哪里、出了问题该看哪里”。
