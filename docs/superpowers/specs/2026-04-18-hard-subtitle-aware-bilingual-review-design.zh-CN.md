# 硬字幕感知的双语审片导出设计

## 目标

减少“中英双语审片版”在原片已带硬中文字幕时的理解歧义，让用户始终清楚：

- 自己选择的是审片目标，而不是固定的字幕合成方式
- 系统为什么在导出时提醒
- 最终导出到底采用了哪种策略

## 问题定义

当前产品把“中英双语审片版”同时表达成：

- 用户意图：我要一个适合审片和对照的版本
- 系统实现：保留中文并再烧录一份中英双语字幕

当原片本身已经有硬中文字幕时，这两个层次会冲突：

- 用户看到“中英双语审片版”，容易理解成“系统会智能给我一个可审片版本”
- 当前实现实际会在原片画面上继续烧录一份中文字幕和英文字幕
- 对用户来说结果像是“系统重复压了中文字幕”，既遮挡画面，也破坏信任

问题不在于字幕技术本身，而在于产品没有把“目标”和“导出策略”拆开表达。

## 核心决策

### 1. 把“审片目标”和“导出策略”拆成两层心智

用户选择的仍然是审片目标，但导出时采用哪种字幕策略，由系统在有足够证据时再明确提示。

- 目标：`双语审片`
- 策略 A：`保留原字 + 补英文`
- 策略 B：`清理原字 + 重做双语`

这样可以避免用户在创建任务时过早做技术决策。

### 2. 只在导出时介入，不在创建任务时新增复杂度

创建任务阶段不新增“原片是否有字幕”“是否保留原字幕”等表单项。

理由：

- 创建任务时，用户通常还不清楚系统对原片硬字幕的判断结果
- 过早暴露技术分叉，会让“成品目标”失去简洁性
- 只有在导出时，系统才拥有 OCR 检测结果、素材状态和更可靠的判断依据

### 3. 检测到硬中文字幕时，默认推荐“保留原字 + 补英文”

这是本次交互的默认推荐策略。

理由：

- 它最符合“少返工、少打扰、快速审片”的主目标
- 用户已经接受原片画面中的中文存在，系统不应默认推动更重的擦字幕流程
- 从结果上最容易理解：原片中文照旧，系统补一层英文供对照

### 4. 不做静默自动切换

即使系统已经检测到原片有硬中文字幕，也不能静默改变双语导出行为。

必须显式提示用户：

- 系统检测到了什么
- 为什么会提示
- 当前推荐采用哪种策略
- 如果用户需要标准化输出，也可以改走“清理原字 + 重做双语”

## 用户体验方案

### 创建任务页

#### 成品目标名称

建议把 `中英双语审片版` 调整为 `双语审片版`。

理由：

- “中英双语”会让用户自然联想到“系统一定会再烧一份中文”
- “双语审片”强调的是场景和用途，不提前绑定实现方式

#### 卡片说明文案

建议文案：

`适合审片和对照。导出前系统会检测原片是否已带中文字幕，并推荐合适的双语方式。`

系统能力标签建议保留“适合审片”，弱化“优先 OCR 字幕”这种实现导向词汇。

### 导出抽屉

#### 触发条件

当满足以下条件时，在用户点击“导出成品”后展示强提示：

- 当前导出 profile 为 `双语审片`
- 检测结果显示原片存在较稳定的硬中文字幕

#### 强提示内容

标题：

`检测到原片已有中文字幕`

说明：

`如果继续烧录中文，可能出现重复或遮挡。建议保留原片中文字幕，只补英文对照。`

#### 策略选择

主选项，默认选中：

- `推荐：保留原字 + 补英文`
- 说明：`保留原片中的中文字幕，在另一行补英文，适合快速审片和对照。`

次选项：

- `清理原字 + 重做双语`
- 说明：`先使用干净画面，再由系统统一烧录中英双语字幕，适合需要标准化审片版本时使用。`

#### 操作按钮

- 主按钮：`按推荐导出`
- 次按钮：`使用重做双语`
- 弱操作：`取消`

### 导出后的结果说明

导出完成后，任务详情和导出摘要必须明确展示：

- `导出目标：双语审片`
- `导出策略：保留原字 + 补英文`

或：

- `导出目标：双语审片`
- `导出策略：清理原字 + 重做双语`

避免用户看到结果后仍不清楚系统到底做了什么。

## 检测模型

### 检测来源

本次不引入新的独立检测链路，优先复用现有 OCR 产物。

系统可基于 `ocr-detect` 阶段结果判断是否存在硬中文字幕：

- `ocr_events.json`
- `detection.json`
- `ocr_subtitles.source.srt`

### 判断分级

为了避免误判造成强干扰，检测结果分成三档：

- `none`
  没有明显硬字幕证据，不展示强提示
- `possible`
  存在一定字幕迹象，展示轻提示，但不阻断导出
- `confirmed`
  字幕证据稳定，展示强提示并要求用户明确选择策略

本次实现允许先只落地 `none / confirmed` 两档，只要接口结构为后续扩展留口即可。

## 产品行为映射

### 双语审片 + 未检测到硬中文字幕

- 默认按现有双语导出走
- 不弹策略选择
- 导出摘要显示 `标准双语导出`

### 双语审片 + 检测到硬中文字幕 + 用户选“保留原字 + 补英文”

- 使用原始视频作为底板
- 仅烧录英文字幕
- 不再额外烧录一条中文字幕
- 导出摘要显示 `保留原字 + 补英文`

### 双语审片 + 检测到硬中文字幕 + 用户选“清理原字 + 重做双语”

- 优先要求有 clean video，或可通过现有模板补跑得到
- 使用 clean video 作为底板
- 烧录系统生成的中文字幕与英文字幕
- 导出摘要显示 `清理原字 + 重做双语`

如果 clean video 不可用，应明确提示用户先补跑擦字幕链路，而不是静默回退。

## 前后端数据建议

### 前端需要新增的概念

- `hardSubtitleStatus`
  - `none`
  - `confirmed`
- `bilingualExportStrategy`
  - `auto_standard_bilingual`
  - `preserve_hard_subtitles_add_english`
  - `clean_video_rebuild_bilingual`

### 后端需要具备的能力

- 对任务暴露硬字幕检测状态
- 在导出接口中接收双语导出策略
- 在导出结果和持久化的 delivery config 中记录实际采用的策略

## 文案原则

- 不向用户暴露 `OCR`、`ASR`、`subtitle_source`、`video_source` 这类底层实现词
- 始终先讲用户意图，再讲系统建议
- 文案避免暗示“系统已经帮你做了决定”
- 推荐文案必须解释推荐的原因

## 不在本次范围

- 不新增创建任务页的字幕检测配置项
- 不做自动静默切换
- 不在本次实现中重做整个导出抽屉结构
- 不解决所有复杂字幕场景，如片头片尾字幕分段存在、双语原生硬字幕、多语言混排等

## 验收标准

- 用户在创建任务时不再误以为“双语审片”一定会重新压一份中文字幕
- 当原片有硬中文字幕时，导出前会出现清晰、可理解的策略选择
- 默认推荐为“保留原字 + 补英文”
- 选择推荐策略后，最终导出不会再重复烧录中文字幕
- 导出完成后，用户能在任务详情中明确看到采用的策略
