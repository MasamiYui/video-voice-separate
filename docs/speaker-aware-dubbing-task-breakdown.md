# 说话人感知配音能力任务拆解

- 项目: `video-voice-separate`
- 文档状态: Draft v2
- 创建日期: 2026-04-11
- 关联文档: [speaker-aware-dubbing-plan.md](/Users/masamiyui/OpenSoureProjects/Forks/video-voice-separate/docs/speaker-aware-dubbing-plan.md)

## 1. 目标

这一版任务拆解按 **核心功能优先** 来排，不再把流水线、协议、CLI 骨架放在最前面。

原则是:

- 先把单个能力做成、做稳、做可测
- 每个能力都能独立输入、独立输出、独立验收
- 能力做通以后，再去做统一协议、串联调度和工程封装

目标能力仍然是这 5 类:

1. 识别“说了什么”
2. 识别“谁在什么时候说”
3. 识别“这个人是谁”，并可跨视频复用
4. 把中文改成适合英文配音的脚本
5. 用原说话人风格生成英文配音

## 2. 总体策略

建议先做 **功能件**，后做 **总装件**。

这里的“功能件”指单个核心能力:

- 说话人归因转写
- 声纹建档与检索
- 翻译脚本生成
- 声音克隆与英文合成
- 时间贴合与混音

这里的“总装件”指:

- 统一目录协议
- manifest
- CLI 串联
- 可恢复运行
- 阶段缓存

因此，实施顺序建议改为:

`A -> B -> C -> D -> E -> F`

其中:

- `A` 到 `E` 是核心功能
- `F` 才是最后的流水线整合

## 3. 任务总览

1. 任务 A: 说话人归因转写
2. 任务 B: 声纹建档与说话人检索
3. 任务 C: 面向配音的英文脚本生成
4. 任务 D: 声音克隆与英文单人合成
5. 任务 E: 多说话人时间贴合与混音
6. 任务 F: 流水线与工程封装

## 4. 任务拆解

## 4.1 任务 A: 说话人归因转写

### 目标

先把最基础也最关键的能力做出来:

- 输入一条中文人声音频
- 输出带说话人标签和时间戳的中文转写

这一步不要求跨视频 speaker 命名稳定，只要求在当前样本内回答:

- 说了什么
- 谁在说
- 从什么时候开始，到什么时候结束

### 输入

- `voice.wav` 或 `voice.mp3`

### 输出

- `segments.zh.json`
- 可选 `segments.zh.srt`

### 完成定义

- 每条句段至少包含:
  - `start`
  - `end`
  - `text`
  - `speaker_label`
- 单人样本可以稳定识别为单 speaker
- 多人样本可以分出多个 speaker_label

### 验收标准

- 单人样本中 speaker 数量不能明显错误
- 双人样本中大部分句段 speaker 归因正确
- 时间戳合法，不能出现大面积交叉和倒序
- 输出文本能用于后续人工编辑

### 测试方式

- 单元测试:
  - JSON schema 校验
  - 时间戳合法性校验
  - `speaker_label` 非空校验
- 集成测试:
  - 单人样本 1 个
  - 双人样本 1 个
  - 检查 speaker 数量、句段数量、时间轴覆盖率
- 半自动验收:
  - 人工抽查 20 到 30 条句段

### 当前阶段定位

- 这是第一优先级
- 后续所有任务都依赖它

## 4.2 任务 B: 声纹建档与说话人检索

### 目标

把任务 A 里的临时标签 `SPEAKER_00`、`SPEAKER_01` 升级成可复用身份。

重点先做这两个能力:

- 从一段参考音频里提取 speaker embedding
- 用 embedding 检索或匹配已有 speaker 库

这一步先不追求复杂产品交互，只要先把“建档”和“匹配”能力做对。

### 输入

- 某个 speaker 的参考音频片段
- 可选已有 speaker 库

### 输出

- `speaker_profile.json`
- `speaker_registry.json`
- speaker 相似度匹配结果

### 完成定义

- 能为一个 speaker 生成稳定 embedding
- 能把新 speaker 与已有 registry 做相似度匹配
- 能输出 top-k 候选和分数

### 验收标准

- 同一人物的不同音频输入，相似度明显高于不同人物
- 同一人物跨两个样本时，top-1 匹配基本稳定
- speaker registry 可新增和读取

### 测试方式

- 单元测试:
  - embedding 维度校验
  - embedding 归一化校验
  - registry 读写测试
- 集成测试:
  - 同一说话人两段音频
  - 不同说话人两段音频
  - 比较同人和异人的相似度差异
- 半自动验收:
  - 人工检查 top-1 匹配和阈值是否合理

### 当前阶段定位

- 这是第二优先级
- 先把 speaker embedding 能力做成，再反向接回任务 A 的 speaker 标签稳定化

## 4.3 任务 C: 面向配音的英文脚本生成

### 目标

先把“可配音英文文案”做出来，而不是直接做最终 TTS。

这一步要解决的是:

- 中文原句和英文句段的一一映射
- 英文文案适合口播
- 专有名词、品牌名、数字表达可控

### 输入

- `segments.zh.json`
- 可选术语表
- 可选品牌词表

### 输出

- `translation.en.json`
- `translation.en.editable.json`

### 完成定义

- 每条英文句段都能映射回原中文句段
- 支持人工修订英文文案
- 修订后的文案可以直接给 TTS 使用

### 验收标准

- 映射关系稳定，不丢句段
- 术语和品牌名可按规则替换
- 修改单条英文文案后，不需要回退重做 ASR 或声纹步骤
- 文案整体更像口播脚本，而不是字面对译

### 测试方式

- 单元测试:
  - JSON schema 校验
  - 原文到译文映射测试
  - 术语替换规则测试
- 集成测试:
  - 选一段包含口语、数字、品牌名的中文样本
  - 检查输出结构完整且可编辑
- 半自动验收:
  - 人工审阅若干句段，确认英文更适合配音

### 当前阶段定位

- 这是第三优先级
- 它是 TTS 之前必须先稳定的文本层

## 4.4 任务 D: 声音克隆与英文单人合成

### 目标

先把“单个 speaker 的英文声音克隆”做成，不一开始就攻克整段多说话人视频。

也就是说，先验证这件事本身成立:

- 给一个说话人的参考音频
- 给一段英文文本
- 生成一段可听、可懂、音色接近原 speaker 的英文语音

### 输入

- 单个 speaker 的参考音频
- 单条或多条英文文本

### 输出

- `speaker_demo_en.wav`
- 可选逐句生成的音频片段

### 完成定义

- 单 speaker 可稳定生成英文音频
- 音色在多句生成中保持基本一致
- 音频没有明显爆音、截断、全静音等错误

### 验收标准

- 输入同一 speaker 的不同英文文本，输出音色不明显漂移
- 不同 speaker 的输出能听出明显差异
- 英文可懂度达到可人工审核的水平

### 测试方式

- 单元测试:
  - 参考音频合法性校验
  - TTS 输入结构校验
  - 输出音频文件存在性和时长校验
- 集成测试:
  - 单 speaker 样本至少 2 条英文文本
  - 两个不同 speaker 的对照测试
- 半自动验收:
  - 人工试听自然度、音色一致性和可懂度

### 当前阶段定位

- 这是第四优先级
- 必须先把“单 speaker 可用”证明出来，再去做多 speaker 拼装

## 4.5 任务 E: 多说话人时间贴合与混音

### 目标

在任务 D 已经可用的前提下，再去解决“多 speaker、按时间线放回去”的问题。

这一步不再研究声纹或翻译本身，而是解决:

- 每句英文配音放到哪里
- 多 speaker 怎么拼接
- 与 background 怎么混

### 输入

- 多 speaker 英文音频片段
- 原始中文句段时间线
- `background.wav`

### 输出

- `dubbed_voice_en.wav`
- `dubbed_mix_en.wav`
- 可选 `dubbed_mix_en.mp3`

### 完成定义

- 每条英文句段能对齐原时间窗口
- 多个说话人的片段能正确拼接
- 与背景混音后能形成可试听成品

### 验收标准

- 没有明显前后错位
- 没有大面积句段覆盖冲突
- 成品时长与原始节奏接近
- 人声和背景电平关系基本合理

### 测试方式

- 单元测试:
  - 时间贴合规则测试
  - 混音参数测试
  - 输出格式测试
- 集成测试:
  - 选一个多人短样本跑完整贴合和混音
  - 校验成品时长是否在可接受范围
- 半自动验收:
  - 人工试听完整短样本

### 当前阶段定位

- 这是第五优先级
- 它依赖前面文本和 TTS 已基本可用

## 4.6 任务 F: 流水线与工程封装

### 目标

把前面已经验证通过的单功能串起来，形成真正可复用的工程产品。

这一步才处理:

- 统一目录结构
- manifest
- 阶段缓存
- CLI 编排
- 可恢复运行
- 报告输出

### 输入

- 任务 A 到 E 的能力模块

### 输出

- 统一 CLI
- 标准目录结构
- `manifest.json`
- `report.json`

### 完成定义

- 可以按阶段运行，也可以整链路运行
- 各阶段错误能被定位
- 中间结果可复用，不必每次全量重跑

### 验收标准

- 同一输入能稳定跑出完整目录结构
- 支持单阶段调试
- 支持重跑某一阶段而不破坏其他阶段结果
- 错误日志足以定位失败阶段

### 测试方式

- 单元测试:
  - manifest schema 校验
  - 阶段状态机测试
  - 缓存命中规则测试
- 集成测试:
  - 用一个短样本跑通完整链路
  - 中途删除一个阶段产物后验证可恢复运行

### 当前阶段定位

- 这是最后优先级
- 只有前面核心能力已经证明成立，做这一步才有意义

## 5. 推荐实施顺序

新的建议顺序是:

1. 先做任务 A，拿到稳定的说话人归因转写
2. 再做任务 B，把 speaker embedding 和 registry 跑通
3. 接着做任务 C，把英文脚本生成做成可编辑能力
4. 再做任务 D，验证单 speaker 英文声音克隆
5. 然后做任务 E，解决多 speaker 拼装和混音
6. 最后做任务 F，把前面能力产品化

## 6. 里程碑定义

为了让每个阶段都有明确产出，建议只保留 3 个里程碑。

### 里程碑 1: 文本与 speaker 能力成立

包含:

- 任务 A
- 任务 B

验收结果:

- 能回答“谁在什么时候说了什么”
- 能初步回答“这个人是不是已知 speaker”

### 里程碑 2: 英文脚本与单人配音能力成立

包含:

- 任务 C
- 任务 D

验收结果:

- 能生成可编辑英文配音脚本
- 能针对单个 speaker 生成可用英文语音

### 里程碑 3: 成品能力成立

包含:

- 任务 E
- 任务 F

验收结果:

- 能生成完整英文混音成品
- 能用统一工程入口重复运行

## 7. 当前建议

如果按“核心功能优先”的原则推进，下一步应该只做一件事:

- 先实现 **任务 A: 说话人归因转写**

原因很直接:

- 它是全链路的最小闭环输入
- 它能最快验证 ASR 和 diarization 选型是否靠谱
- 它的结果会直接决定后面 speaker registry、翻译和 TTS 的数据结构

所以，下一阶段的首个代码目标应当是:

`voice.wav -> segments.zh.json (带时间戳和 speaker_label)`
